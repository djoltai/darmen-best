# darmen.best — путеводитель проекта

Это репозиторий сайта darmen.best и лабораторных интерактивов под `/lab/*`.
Файл стабильный, меняется редко. Адаптирован из bundle'а
`projects/labs/design-system/skill/reference/CLAUDE.md` (Cowork).

## Что это за сайт

darmen.best — персональный сайт Дармена. Главная — лендинг с короткой
презентацией и навигацией. Раздел `/lab/*` — интерактивные визуализации
финансовых и вероятностных концепций. Лабы анонсируются и обсуждаются в
постах LinkedIn.

## Стек

- Astro (`^6.1.8`)
- TypeScript для скриптов
- Чистый CSS (Tailwind удалён в коммите `b0d5bfa`, см. JOURNAL 2026-05-18)
- Шрифты через `@fontsource`:
  - **IBM Plex Sans** (sans) — главная и публичные страницы, веса 400/500/600
  - **JetBrains Mono** — числа в kicker, формулы, chart-labels
  - **EB Garamond** — только буква `D` в wordmark главной
  - **Inter** — остался в лабах (`/lab/frm-var`, `/lab/frm-var-v2`),
    миграция на IBM Plex Sans планируется отдельной сессией
- Деплой: статика, билд в `dist/`

**Источник правды для дизайн-системы:** `src/styles/design-system.css`
в репо (production-копия) + `projects/labs/design-system/skill/`
в Cowork (полный bundle с README.md, SKILL.md, мокапами, превью).

**Конвенция лаба:** у каждого лаба свой `<slug>`. Page, scripts, styles
по этому slug'у. В этом репо есть историческая инконсистенция: page
`frm-var.astro` пользуется scripts `frm-vol-drag/` и styles
`frm-vol-drag.css`. Не «улучшать» попутно.

## Проектные файлы памяти

- `CLAUDE.md` (этот файл) — устройство проекта, читать в начале сессии
- `ROADMAP.md` — что в работе, что дальше, что в идеях
- `JOURNAL.md` — append-only журнал сессий, что сделано, что узнали
- `docs/handoffs/` — брифы от Claude Design / Cowork, читать перед
  новой существенной задачей

В начале сессии Claude Code: прочитать `CLAUDE.md`, `ROADMAP.md`,
последние 1-2 записи `JOURNAL.md`. Если задача по конкретному лабу или
поверхности — также читать соответствующий `docs/handoffs/<date>-*.md`.

В конце значимой сессии: добавить датированную запись в `JOURNAL.md`,
обновить `ROADMAP.md` если что-то закрыто/открыто/переехало, прибить
Resolved-блок в handoff'е.

## Контракт нового лаба / новой поверхности

Перед началом реализации должен существовать design-doc или handoff:
либо `docs/handoffs/YYYY-MM-DD-<slug>-*.md`, либо явный бриф в чате.
Если его нет — попроси создать сначала, не приступай к коду.

Минимальное содержание design-doc: что объясняем, ключевая интуиция,
состояния интерактива, ключевые формулы, acceptance criteria.

## Дизайн-система

См. `src/styles/design-system.css` и README.md в Cowork bundle'е.
Сводка хард-правил:

- Эмодзи в публичной части — ноль. Везде.
- Heading-вес 500, никаких 700. Bold-display — AI-tell.
- Фон везде `#FAF6E8`. Тёмный `#1A1A1A` только в драматических моментах
  внутри лабов.
- Акценты: teal `#07434B` для ссылок/положительного/бренда, coral
  `#C75B4F` только для риска/потерь/отрицательного, gold `#C9A443`
  один раз во всей системе (финал Volatility Drag).
- Тире длинное, минус — настоящий `−` (U+2212), не дефис.
- Никаких теней, градиентов, скруглений > 3px, иконочных шрифтов.
- Если запрос ломает хард-правило — флаг, три пути (one-off / update
  system / alternative in-system), ждать выбора (см. SKILL.md в Cowork).

## Тестирование

Тесты обязательны для математических ядер (`scripts/lab/<slug>/calc.ts`,
`distributions.ts`), где правка может незаметно сломать формулу.

**Тестового раннера в репо сейчас нет** (vitest не установлен). Это
осознанный tech-debt сессии 2026-05-18 — handoff требовал теста, но
запрещал новые dep'ы кроме шрифтов. Когда vitest появится по другой
задаче — прикрутить тесты к существующим математическим модулям и к
детерминированному SVG в `LabCard.astro`.

Для визуальной/UX-части — acceptance-чеклист в handoff/design-doc.

## Стиль кода

- TypeScript strict
- Никаких новых внешних зависимостей без явной просьбы
- Match existing style, не «улучшать» соседний код
- Комментарии только когда они объясняют нетривиальное «почему», не «что»

## Workflow

1. Новая задача → план + ожидание апрува
2. Перед удалением/перезаписью/переименованием существующего файла —
   diff и пауза для подтверждения
3. После каждого видимого чанка (новая страница, существенный визуальный
   слой) — пуш в dev, ждать визуальной проверки. Approval ≠ визуальная
   проверка
4. После каждого крупного шага — короткое саммари: что сделано, что дальше
5. Не выходить за рамки `darmen-best/` если не попросили явно
6. В конце задачи — список созданных/изменённых файлов
7. Если сессия рождает артефакт для другого инструмента (пост в Cowork,
   обратный бриф) — финальный handoff-блок

## Карпати-принципы

1. **Think before coding.** Явно проговаривать допущения. Не угадывать
   молчаливо при неоднозначности — спросить. Пушбэкать когда есть простой
   вариант, который пользователь не видит.
2. **Simplicity first.** Минимум кода, решающего задачу. Без спекулятивной
   гибкости, без преждевременных абстракций, без обработки невозможных
   кейсов. 200 строк могут стать 50 — переписать.
3. **Surgical changes.** Касаемся только того, что требует задача. Не
   улучшаем соседний код, форматирование, комментарии. Match existing
   style. Замеченный мёртвый код — упомянуть, не удалять самовольно.
4. **Goal-driven execution.** Превращать императивные задачи в
   проверяемые цели. Сильные критерии успеха позволяют зацикливаться
   независимо.
